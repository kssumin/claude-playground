## EXPLAIN

### 테스트 대역(Test Double) 선택 기준

토스는 테스트 대역을 "최후의 수단"으로 취급한다.

| 상황 | 선택 |
|------|------|
| 외부 서비스 (외부 API, 메시지 큐) | **Fake 객체** 구현 |
| 내부 서비스 (같은 시스템) | **실제 객체** 사용 |
| 내부 서비스 + 부수효과 (이메일, SMS) | **Dummy 객체** 사용 |
| 데이터 확보가 어려운 경우 | **Mock 객체** (최후의 수단) |

### 왜 Mock을 최소화하는가?

Mock을 과도하게 사용하면:
- 구현 세부사항에 결합 → 리팩토링 시 테스트가 깨짐
- 실제 동작을 검증하지 못함
- 테스트가 "통과하지만 버그가 있는" 상태 발생

### Fake vs Mock 차이

```
Fake: 실제 동작하는 가벼운 구현체 (인메모리 저장소 등)
Mock: 호출 여부만 검증하는 껍데기
```

## EXECUTE

프로젝트의 의존성을 분석해본다.

```
[외부 서비스 목록] → Fake 구현 대상
- 예: FCM (Firebase Cloud Messaging)
- 예: 외부 알람 API

[내부 서비스 목록] → 실제 객체 사용
- 예: UserRepository
- 예: AlarmRepository

[부수효과 발생 서비스] → Dummy 사용
- 예: 이메일 발송 서비스
```

## QUIZ

**Q3.** 다음 중 Fake 객체를 사용하기에 가장 적절한 상황은?

A) UserRepository (내부 DB 접근)
B) FCM 푸시 알림 서비스 (외부 서비스)
C) 간단한 계산 로직
D) 로깅 서비스

**정답: B** - 외부 서비스는 Fake로 대체해 테스트 안정성을 확보한다.
