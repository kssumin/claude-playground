## EXPLAIN

### 토스의 테스트 3분류

토스는 테스트를 목적에 따라 3가지로 나눈다.

#### 1. 도메인 정책 테스트 (단위 테스트)
- 도메인 객체의 비즈니스 규칙 검증
- **실제 객체를 우선 사용** (Mock 최소화)
- 경계값 중심의 단순한 테스트

```java
// 예시: 알람 발송 도메인 정책
@Test
void 알람은_수신_동의한_사용자에게만_발송된다() {
    User user = new User(NotificationConsent.DENIED);
    assertThat(alarm.canSendTo(user)).isFalse();
}
```

#### 2. 유스케이스 테스트 (통합/인수 테스트)
- 실제 HTTP 요청부터 DB까지 전 계층을 커버
- `@AcceptanceTest`로 스프링 컨텍스트 재사용 → 속도 최적화
- JSON 기반 테스트 데이터 셋업
- **블랙박스 테스트**: 내부 구현이 아닌 비즈니스 동작만 검증

#### 3. 직렬화/역직렬화 테스트
- API 스펙 변경 감지
- DTO ↔ JSON 변환 정확성 검증

## EXECUTE

현재 프로젝트에서 각 분류에 해당하는 테스트 후보를 나열해본다.

```
[도메인 정책 테스트 후보]
- 예: 알람 발송 조건 검증

[유스케이스 테스트 후보]
- 예: 알람 등록 API E2E

[직렬화 테스트 후보]
- 예: AlarmRequest JSON 파싱
```

## QUIZ

**Q2.** 유스케이스 테스트에서 블랙박스 테스트를 선택하는 이유는?

A) 코드가 더 짧아서
B) 내부 구현 변경에 영향받지 않고, 비즈니스 동작 변경에만 테스트가 깨지도록
C) Mock 사용이 어려워서
D) 스프링 컨텍스트 재사용이 안 되어서

**정답: B**
