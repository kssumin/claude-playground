## EXPLAIN

### 프로젝트에 적용하기 — 실전 체크리스트

토스 전략을 내 프로젝트에 적용하는 순서:

#### Step 1. 테스트 우선순위 분류
```
HIGH (반드시 테스트)
- 결제, 정산, 알람 발송 등 핵심 비즈니스
- 장애 시 사용자 영향이 큰 기능

MEDIUM (여유 되면 테스트)
- 조회 API, 설정성 기능

LOW (수동 테스트로 대체)
- 이벤트성 임시 기능
- 단기 수명 코드
```

#### Step 2. 테스트 종류 결정
```
도메인 객체에 비즈니스 규칙이 있는가?
  → YES: 도메인 정책 테스트 (단위 테스트)

여러 계층을 통합 검증해야 하는가?
  → YES: 유스케이스 테스트 (통합 테스트)

API 스펙 변경 감지가 필요한가?
  → YES: 직렬화 테스트
```

#### Step 3. 테스트 대역 결정
```
의존성이 외부 서비스인가?
  → Fake 구현

의존성이 내부 서비스인가?
  → 실제 객체 사용

부수효과가 있는가?
  → Dummy 사용

데이터 준비가 너무 복잡한가?
  → Mock (최후의 수단)
```

## EXECUTE

현재 프로젝트의 테스트 전략 문서를 작성해본다.

```markdown
# 테스트 전략

## 핵심 기능 (반드시 테스트)
- [ ] 기능 1:
- [ ] 기능 2:

## 테스트 종류 계획
- 도메인 정책 테스트:
- 유스케이스 테스트:
- 직렬화 테스트:

## 테스트 대역 계획
- Fake:
- Dummy:
- 실제 객체:
```

## QUIZ

**Q4 (최종).** 토스 테스트 전략의 핵심 원칙을 한 문장으로 표현하면?

A) 모든 코드에 100% 테스트 커버리지를 달성한다
B) 비즈니스 가치가 높은 부분에 선택적으로, 여러 계층을 통합 검증하는 테스트를 작성한다
C) Mock을 적극 활용해 의존성을 제거한다
D) 단위 테스트만으로 모든 것을 검증한다

**정답: B**
