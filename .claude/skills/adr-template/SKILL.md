# ADR 작성 가이드

## 트리거 조건

- `/design` Step 7 (설계 리뷰 통과 후)
- 사용자가 "ADR 작성", "ADR 만들어줘" 요청 시

## 파일 규칙

- 경로: `docs/adr/ADR-{번호}-{kebab-case-제목}.md`
- 기존 번호 확인: `ls docs/adr/ 2>/dev/null | sort -V | tail -1`

---

## ADR의 경계

ADR은 **"무엇을 왜 결정했는가"**를 기록하는 문서다.

**ADR에 넣는 것**: 결정의 맥락, 근거, 대안, 트레이드오프, 파생 효과, 리스크 방향.
**ADR에 넣지 않는 것**: 클래스/메서드 시그니처, DDL 구문, Kafka 메시지 필드, 테스트 케이스 목록, 모듈별 변경 상세 → 별도 구현 계획서(`/plan`)에서 다룸.

## 섹션 구성과 각 역할

### Status
`Proposed` | `Accepted` | `Deprecated` | `Superseded by ADR-{N}` 중 하나. 이 결정의 현재 상태.

### Date
작성일.

### Deciders
이 결정에 참여한 사람.

### Context — "왜 이 결정이 필요한가?"

세 파트로 구성:

- **문제**: 현재 뭐가 안 되는지 한 줄로.
- **왜 지금**: 예방적인지 대응적인지, 지금 해야 하는 이유. 예방적이면 "예방적이다" 명시 + 규모 추정에서 도출된 문제 시나리오. 대응적이면 증거(로그, p99 지표) 인용. 리뷰어의 "지금 안 해도 되는 거 아니야?"에 답할 수 있어야 한다.
- **규모 추정**: 수치로 문제의 크기를 보여줌. 이게 있어야 Decision의 근거가 설득력을 가진다.

### Decision — "무엇을 왜 결정했는가?"

ADR의 핵심. 각 결정마다 세 가지를 적는다:

- **결정**: 한 줄로 뭘 하는지.
- **근거**: 왜 이렇게 하는지 (규모 추정, 트레이드오프, 과거 장애 전례).
- **주요 영향**: 이 결정 때문에 뭐가 바뀌는지 (스키마 방향, API 호환성, 운영 모델). 단, 구현 상세(클래스명, 메서드 시그니처)는 적지 않는다.

공통 전제가 있으면 Decision 서두에 한 번 적고 개별 결정에서 반복하지 않음.
하나의 결정 안에 여러 관심사가 뒤섞이면, 하위 결정으로 위계를 나눈다 (예: "파티션 도입 → 스키마 영향 → 운영 메커니즘").

### Alternatives Considered — "다른 선택지는 왜 기각했는가?"

- 선택안 포함 모든 옵션을 **동일 기준**으로 비교표 작성.
- **선택안의 약점도 동등한 무게로** 서술 — 약점을 가볍게 쓰면 리뷰어 신뢰 하락.
- 기각 사유 뒤에 "선택안의 비용" 별도 서술.

### Consequences — "이 결정을 실행하면 어떤 파생 효과가 생기는가?"

Decision의 근거를 반복하는 게 아니라, 결정 이후에 생기는 **2차 영향**을 적는다:

- **Positive**: 이 결정 덕분에 앞으로 가능해지는 것.
- **Negative**: 이 결정 때문에 앞으로 제약되는 것.

### Risks — "뭐가 잘못될 수 있고, 어떻게 대응하는가?"

5개 이상이면 두 카테고리로 나눈다:

- **구현 시 검증 필수**: 실제로 시간 써서 확인해야 하는 것. 번호 + [높음/중간] 태그.
- **인지 사항**: 알고는 있되 지금 행동할 건 없는 것. 번호 + 대시(—)로 대응 방향만 한 줄.

경합/레이스 컨디션은 시나리오 + **왜 수용 가능한지** 근거 명시.
스키마 변경이 있으면 마이그레이션 전략을 Risks에 포함.

### Performance Targets — API 변경 시

목표 수치 + 그 수치의 근거 + 미달 시 대응. 근거 없는 목표는 의미가 없으니 "왜 이 숫자인지" 반드시 적는다. 가정이면 "직관적 추정이며 실측 데이터 없음" 명시.

미달 시 대응은 "ADR 범위 내 시도" vs "별도 ADR 필요" 경계 구분.

### Implementation — 별도 구현 계획서 링크

ADR에 구현 상세를 넣지 않는다. 한 줄 링크로 끝:

```
→ [implementation-plan-{번호}.md](./implementation-plan-{번호}.md) 참조
```

## 작성 체크리스트 (커밋 전)

- [ ] Context에 문제 / 왜 지금 / 규모 추정 세 파트가 있는가?
- [ ] 각 결정이 "결정 → 근거 → 주요 영향" 구조인가?
- [ ] 공통 전제가 있다면 서두에 한 번만 언급했는가?
- [ ] Alternatives에서 선택안의 약점이 동등한 무게로 서술되었는가?
- [ ] Consequences가 Decision의 근거를 반복하지 않고 파생 효과/제약을 다루는가?
- [ ] 모든 수치에 근거가 있는가? (가정이면 "가정임" 명시)
- [ ] 레이스 컨디션이 있다면 시나리오 + 수용 근거가 있는가?
- [ ] Risks가 5개 이상이면 "구현 시 검증 필수" / "인지 사항"으로 나눴는가?
- [ ] 스키마 변경이 있다면 마이그레이션 전략이 Risks에 있는가?
- [ ] Performance Targets에 근거 + 미달 시 대응이 있는가?
- [ ] Implementation이 별도 구현 계획서 링크인가? (ADR에 구현 상세 없음)
