# 설계 전략 문서 작성

## 핵심 철학

> 전략 문서는 **"왜 이렇게 했고, 뭘 못 막는지"**를 설명하는 문서다.
> 구현 디테일(클래스명, 메서드명, 설정값)은 코드가 담당한다. 문서는 판단과 근거를 담는다.

## 워크플로우

```
Step 1: 컨텍스트 수집 (Silent)
Step 2: 문서 구조 설계
Step 3: 초안 작성
Step 4: 상호 링크
Step 5: 검증
```

### Step 1: 컨텍스트 수집

기존 문서와 코드를 읽어 현재 상태를 파악한다. 사용자에게 별도 보고하지 않는다.

- 관련 내용이 흩어진 기존 문서 (architecture.md, ADR, design-considerations.md 등)
- 해당 기능의 실제 구현 코드 (흐름 이해 목적, 문서에 코드를 넣기 위함이 아님)
- 기존 문서의 톤과 스타일

### Step 2: 문서 구조 설계

**필수 흐름: 문제 → 해결(근거) → 미해결**

각 문제를 독립 섹션으로 분리하고, 섹션 제목만 읽어도 "이 문제는 이걸로 해결했다"가 보여야 한다.

```
## 1. {문제 A} → {해결책 A}
### 왜 발생하는가
### 해결
### 근거

## 2. {문제 B} → {해결책 B}
### 왜 발생하는가
### 해결
### 근거

## N. 추가 고려 사항: {미해결 문제}
### 현황 / 문제점
### 올바른 방향
### 향후 구현 방향

## 요약
### 방어 테이블
### 트레이드오프 테이블
### 관련 문서
```

### Step 3: 초안 작성

**서술 규칙**

| 규칙 | 설명 |
|------|------|
| 개념으로 말한다 | 클래스명, 메서드명, 설정 키 대신 "무엇을 하는지"를 서술 |
| 흐름은 다이어그램으로 | Mermaid sequenceDiagram 또는 이미지로 큰 그림 제공 |
| 비교는 표로 | 시나리오 비교, 트레이드오프, 방어 범위 등 |
| 막는 것 / 못 막는 것을 구분한다 | 방어 메커니즘의 한계를 명시적으로 보여준다 |
| 근거를 붙인다 | "왜 이 선택인가"를 논리적 추론 또는 업계 관행으로 뒷받침 |
| 겹치면 줄인다 | 앞에서 설명한 내용을 뒤에서 반복하지 않고 "앞서 정리한 대로" 연결 |
| 문서의 관심사만 남긴다 | 이 문서가 다루는 주제와 무관한 구현 세부사항은 제거 |

**금지 사항**

- NEVER: 코드 블록으로 구현 보여주기 (코드는 코드베이스에 있다)
- NEVER: `@ConfigurationProperties`, `@Component` 등 프레임워크 어노테이션 언급
- NEVER: 특정 클래스/메서드 시그니처 나열
- NEVER: 해결이 완벽한 것처럼 쓰기 (못 막는 범위를 반드시 명시)

**요약 섹션 필수 요소**

1. **방어 테이블**: 단계 | 문제 원인 | 방어 지점 | 메커니즘 | 동작
2. **트레이드오프 테이블**: 결정 | 선택 | 대안 | 수용한 비용
3. **관련 문서 링크**: 이 문서와 연관된 기존 문서 목록

### Step 4: 상호 링크

새 문서를 만들었으면, 관련 기존 문서에도 역방향 링크를 추가한다.

- 기존 문서에서 해당 주제를 다루는 섹션에 `→ [상세](new-doc.md)` 추가
- ADR에 관련 전략 문서 참조 추가

### Step 5: 검증

| 체크 | 질문 | 통과 기준 |
|------|------|----------|
| 흐름 | 문제 → 해결 → 미해결 순서인가? | 섹션 제목만 읽어도 전체 흐름이 보임 |
| 코드 부재 | 코드 블록이 있는가? | 없어야 함 (다이어그램은 OK) |
| 방어 범위 | 막는 것과 못 막는 것이 구분되어 있는가? | 표 또는 명시적 구분 존재 |
| 근거 | 모든 결정에 "왜"가 붙어있는가? | 근거 없는 결정 없음 |
| 겹침 | 같은 내용이 두 번 이상 나오는가? | 나오면 하나로 합치고 참조 |
| 링크 | 기존 문서에 역방향 링크를 추가했는가? | 관련 문서 모두 상호 링크 |
| 충돌 | 기존 문서와 내용이 모순되는가? | 모순 있으면 기존 문서도 수정 |

## 참고: 완성된 문서 예시 구조

`deduplication-strategy.md`에서 적용된 패턴:

```
# 중복 발송 방지 전략

## 1. 클라이언트 중복 요청 → Idempotency-Key + DB UNIQUE
  왜 발생 → 해결 (다이어그램) → 근거

## 2. Kafka 재전달 → Redis 중복 체크
  왜 발생 → 막는 것/못 막는 것 구분 (표)
  → 전제 (at-least-once 수용 근거) → 해결 (다이어그램)
  → 근거 1 (순서) → 근거 2 (원자성)

## 3. Redis 장애 시 발송 정책
  현재 문제점 → 올바른 방향 → 향후 구현

## 4. 요약
  방어 테이블 / 트레이드오프 테이블 / 관련 문서
```
