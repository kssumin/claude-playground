# troubleshooting-doc — 트러블슈팅 경험 문서화

> **핵심 원칙**: 문제를 해결한 경험은 그 자리에서 기록한다.
> 증거 없이 해결한 문제는 다음에 또 겪는다.

---

## Phase 1: 번호 확인

```bash
ls docs/troubleshooting/ 2>/dev/null | grep -E '^TS-[0-9]+' | sort -V | tail -1
```

다음 번호를 채번한다. (TS-001, TS-002, ...)
`docs/troubleshooting/` 디렉토리가 없으면 생성한다.

---

## Phase 2: 대화에서 추출할 정보

| 항목 | 설명 | 필수 |
|------|------|------|
| **증상** | 언제, 어떤 이상 현상이 관찰됐는가 | ✅ |
| **수집한 증거** | 로그, 메트릭, 에러 메시지, 재현 조건 | ✅ |
| **근본 원인** | 왜 발생했는가 (표면 원인 말고 진짜 원인) | ✅ |
| **조사 과정** | 어떤 가설을 세우고 어떻게 검증했는가 | ✅ |
| **해결 방법** | 무엇을 바꿨는가 (코드/설정/인프라) | ✅ |
| **해결 확인** | 어떻게 해결됐음을 확인했는가 | ✅ |
| **예방책** | 재발 방지를 위해 무엇을 할 수 있는가 | 선택 |
| **영향 범위** | 어떤 서비스/사용자에 영향이 있었는가 | 선택 |

## Phase 3: 문서 템플릿

```markdown
# TS-{번호}: {한 줄로 요약한 제목}

## 기본 정보
- **날짜**: {YYYY-MM-DD}
- **심각도**: Critical / High / Medium / Low
- **영향 범위**: {서비스명, 기능, 사용자 범위}
- **해결 소요 시간**: {시간}

## 증상
{언제부터, 무엇이 이상했는가. 관찰 가능한 현상 중심으로.}

## 수집한 증거

### 로그 / 에러 메시지
```
{실제 에러 로그나 스택 트레이스}
```

### 메트릭 / 지표
{CPU, 메모리, 응답시간, 에러율 등 수치가 있으면 기록}

### 재현 조건
{어떤 상황에서 재현되는가. 재현 불가면 "간헐적" 명시}

## 원인 분석

### 가설 → 검증 과정
1. **가설 1**: {무엇이 원인이라고 생각했는가}
   - 검증: {어떻게 확인했는가}
   - 결과: {맞았는가 / 틀렸는가}

2. **가설 2**: {다음 가설}
   - 검증: {어떻게 확인했는가}
   - 결과: {맞았는가 / 틀렸는가}

### 근본 원인
**{표면 원인이 아닌 진짜 원인}**

{왜 이게 발생했는지. 코드/설정/아키텍처 수준의 근본 원인.}

## 해결 방법

### 즉시 조치 (임시)
{긴급하게 취한 조치가 있으면 기록. 없으면 생략.}

### 근본 해결
{어떤 코드/설정/인프라를 어떻게 바꿨는가}

```diff
// 변경 전
-기존 코드

// 변경 후
+수정된 코드
```

## 해결 확인
{어떻게 해결됐음을 확인했는가. 재현 시도, 메트릭 회복, 로그 정상화 등.}

## 예방책
- [ ] {재발 방지를 위해 할 일 1}
- [ ] {재발 방지를 위해 할 일 2}

## 재사용 가능한 교훈
{이 경험에서 다른 상황에도 적용할 수 있는 일반화된 통찰}
```

## Phase 5: 인덱스 업데이트

`docs/troubleshooting/README.md`에 새 항목을 추가한다. 없으면 자동 생성.

```markdown
| ID | 날짜 | 제목 | 심각도 | 근본 원인 요약 |
|----|------|------|--------|----------------|
| TS-001 | 2026-03-15 | Kafka consumer lag spike | High | max.poll.records 기본값 500이 처리 시간 초과 유발 |
```

## 자동 추천 조건

아래 상황이 감지되면 사용자에게 먼저 제안한다:

```
💡 /troubleshooting 으로 기록해두면 나중에 같은 문제 다시 안 겪어도 돼요. 남길까요?
```

| 상황 | 판단 기준 |
|------|-----------|
| **비자명한 원인 발견** | "아 이게 문제였구나" 순간 — 처음 보면 모를 만한 원인 |
| **삽질 시간 발생** | 원인 찾는 데 30분 이상 소요된 흔적이 대화에 있음 |
| **에러 로그 분석** | 스택 트레이스를 해석하고 원인과 해결책이 명확해졌을 때 |
| **설정 값이 원인** | 기본값 / 잘못된 설정이 문제였고 수정 완료됐을 때 |
| **재현 조건이 특이** | "특정 조건에서만 발생" 패턴 — 다음에 또 발생할 가능성 높음 |
| **인프라/외부 연동 이슈** | Kafka, Redis, DB, 외부 API 관련 이슈 해결 완료 후 |

**추천하지 않는 상황:**
- 단순 오타/컴파일 에러 수정
- 문서에 나온 그대로 따라하면 되는 것
- 이미 `docs/troubleshooting/`에 동일한 이슈가 있는 경우

## 작성 원칙

- **증거 중심**: 로그, 수치, 재현 조건이 없으면 다음에 같은 문제를 또 겪는다
- **근본 원인**: "DB가 느렸다"가 아니라 "N+1 쿼리로 인해 300ms → 3s로 증가했다"
- **가설 검증 과정**: 어떻게 틀린 가설을 제거했는지가 다음 트러블슈팅의 자산이 된다
- **예방책은 액션 아이템**: 체크박스로 작성해 후속 조치 추적 가능하게
- **즉시 기록**: 기억이 선명할 때 작성. 하루 지나면 디테일이 사라진다
