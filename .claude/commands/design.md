# /design - 아키텍처 설계 + ADR 작성

> **전제 조건**: brainstorming 스킬로 요구사항이 명확화된 상태에서 실행한다.
> brainstorming이 선행되지 않았으면, 먼저 brainstorming을 실행하도록 안내한다.

확정된 요구사항을 기반으로 아키텍처를 설계하고, 결정사항을 ADR 문서로 기록합니다.
**구현은 사용자 합의 후에만 진행한다.**

## Usage
```
/design "주문 취소 기능 추가"
/design "결제 시스템을 PG사 연동으로 변경"
/design "Redis 캐시 레이어 도입"
```

---

## Step 1: 요구사항 확인

brainstorming에서 확정된 요구사항을 정리한다.
이미 합의된 내용은 재질문하지 않고, 누락된 부분만 보충한다.

1. 기능 요구사항 (What) - brainstorming 결과에서 가져옴
2. 비기능 요구사항 (성능, 보안, 확장성) - 부족하면 보충 질문
3. 제약 조건 확인
4. 누락된 사항만 사용자에게 질문 (1개씩, 객관식 선호)

---

## Step 3: 시스템 아키텍처 분석

**규모 추정 → 논의 → 결정**의 논리적 흐름으로 진행한다.

### 3-1. 용량/규모 추정 (필수 - 항상 수행)
모든 기능 설계의 출발점. 이 숫자가 이후 아키텍처 결정의 근거가 된다.
- 예상 데이터 규모 (레코드 수, 저장 용량)
- 예상 트래픽 (읽기/쓰기 TPS, 피크 트래픽 배수)
- 데이터 증가 속도 (일/월 단위)

### 3-2. 아키텍처 관심사 도출 (규모 + 기능 특성 기반 논의)
3-1의 숫자와 기능 요구사항을 바탕으로, **어떤 아키텍처 관심사를 신경 써야 하는지** 사용자와 논의한다.
요구사항에서 명시적으로 불필요하다고 했거나, 규모상 해당 없으면 과감히 스킵한다.

논의 대상 후보:
- **동시성 제어**: 같은 자원에 동시 쓰기가 발생하는가?
  → 낙관적 락 / 비관적 락 / 분산 락 비교
- **데이터 파티셔닝**: 단일 DB로 감당 가능한 규모인가?
  → 샤딩 키, 전략, 크로스 샤드 쿼리 대응
- **캐싱**: 반복 조회가 많은가? 변경 빈도는 낮은가?
  → 캐시 전략, TTL, 무효화(CDC/이벤트/TTL)
- **데이터 정합성**: 여러 서비스/모듈 간 트랜잭션이 필요한가?
  → 로컬 트랜잭션 / 2PC / Saga / 이벤추얼 컨시스턴시
- **이벤트/메시징**: 비동기 처리나 모듈 간 디커플링이 필요한가?
  → Kafka 토픽, 멱등성, DLQ
- **장애 대응**: 외부 API/인프라 연동이 있는가?
  → Circuit Breaker, Retry, Fallback, Timeout

### 3-3. 결정 도출
논의 결과를 정리한다:
- 신경 써야 할 항목만: 각각의 **선택한 전략 + 근거** 명시
- 관련 없는 항목은 언급하지 않음 (ADR에 스킵 근거 불필요)
- 이 결정들이 Step 4~5 설계에 어떻게 반영되는지 연결

---

## Step 4: 설계 접근법 제안

### 추론 프레임워크 (제안 전 내부 사고)
접근법을 제안하기 **전에** 아래 질문을 스스로에게 던진다. 사용자에게 보여주지 않되 사고의 근거로 활용한다.

```
[내부 검증 - 제안 전]
□ 이 설계에서 가장 먼저 깨질 곳은 어디인가? (Single Point of Failure)
□ 데이터가 10배가 되면 이 설계가 유지되는가?
□ 가장 빈번한 유스케이스의 쿼리 경로는 최적인가?
□ 도메인 모델이 비즈니스 언어를 충실히 반영하는가?
□ 이 설계를 처음 보는 개발자가 30분 안에 이해할 수 있는가?
□ 테스트하기 어려운 부분은 없는가? 있다면 설계 문제인가?
```

### 제안 형식
1. **2-3가지 접근법 제안** (추천 옵션 표시)
2. 각 접근법의 트레이드오프 분석:

| 기준 | Option A | Option B | Option C |
|------|----------|----------|----------|
| 구현 복잡도 | | | |
| 성능 (읽기) | | | |
| 성능 (쓰기) | | | |
| 확장성 | | | |
| 테스트 용이성 | | | |
| 멀티모듈 적합성 | | | |

3. 모듈별 변경 영향도 분석:
   - domain: 어떤 Entity/Port/Service 추가/변경?
   - infra: 어떤 JPA Entity/Repository 추가/변경?
   - app-api: 어떤 Controller/DTO 추가/변경?
   - common: 공통 코드 변경 필요?
   - client-external: 외부 연동 필요?
4. **인프라 변경 분석** (새 인프라 필요 시):
   - docker-compose.yml에 추가할 서비스 정의
   - 필요한 Docker 이미지 및 버전
   - 포트 매핑, 볼륨, healthcheck 설계
   - application.yml 연결 설정
   - Testcontainers 통합 테스트 계획
5. **추천 옵션 + 추천 근거** 명시
6. 사용자 선택 대기

## Step 5: 상세 설계

사용자가 접근법을 선택하면:

1. 구현 단계 상세화 (정확한 파일 경로, 클래스명, 함수명)
2. **데이터 모델 설계**:
   - ERD (테이블, 컬럼, 관계, 인덱스)
   - 도메인 모델 ↔ JPA Entity 매핑
   - 마이그레이션 전략 (DDL)
3. **API 설계** (엔드포인트, Request/Response):
   - HTTP Method, Path, Status Code
   - Request/Response DTO (필드, 타입, 제약조건)
   - 에러 응답 (에러 코드, 메시지)
4. **동시성/정합성 상세** (Step 3에서 선택한 전략의 구체 구현):
   - 락 적용 대상 (어떤 Entity/테이블?)
   - 충돌 시 처리 (재시도 횟수, 에러 응답)
   - 트랜잭션 경계 설계
5. 에러 처리 전략
6. 테스트 전략
7. **인프라 설계** (새 인프라 필요 시):
   - docker-compose.yml 서비스 정의 (이미지, 포트, 볼륨, healthcheck)
   - infra 모듈 Config 클래스 설계
   - application-local.yml 연결 설정
   - .env.example 환경 변수
8. **성능 목표 정의** (API 추가/변경 시 필수):
   - 엔드포인트별 응답시간 목표 (p95, p99)
   - 목표 TPS (초당 처리량)
   - 허용 에러율
   - 예상 트래픽 패턴 (조회/쓰기 비율)
   - → 구현 완료 후 `/perf-test`로 검증

## Step 6: 자기 검증 (제출 전 품질 게이트)

ADR 작성 **전에** 설계 품질을 검증한다. **모든 항목이 통과해야** ADR을 작성한다.

### 필수 체크리스트
```
[완전성 검증]
□ 모든 API 엔드포인트에 에러 케이스가 정의되었는가?
□ 데이터 모델에 인덱스가 쿼리 패턴에 맞게 설계되었는가?
□ 도메인 모델과 JPA Entity 간 매핑이 명확한가?
□ 트랜잭션 경계가 명시되었는가?

[일관성 검증]
□ 기존 ADR의 결정과 충돌하는 부분이 없는가?
□ 기존 코드 패턴(ApiResponse, DomainException 등)을 따르는가?
□ 모듈 의존성 규칙을 위반하지 않는가? (domain → 순수, 하위→상위 참조 금지)

[설계 원칙 검증] (design-principles 룰 참조)
□ 도메인 모델이 빈약하지 않은가? (getter/setter만 있는 Entity가 아닌가?)
□ Aggregate 경계가 적절한가? (하나의 트랜잭션 = 하나의 Aggregate)
□ Port 인터페이스가 도메인 관점으로 정의되었는가? (인프라 용어 누출 없음)
□ 순환 의존이 없는가?

[현실성 검증]
□ Step 3의 규모 추정이 설계 결정과 실제로 연결되는가? (숫자 → 결정의 인과관계)
□ 과제 전형 범위 내에서 구현 가능한 복잡도인가? (오버 엔지니어링 주의)
□ 테스트 가능한 설계인가? (외부 의존성이 Port로 추상화되었는가?)
```

### 실패 시
- 체크리스트 미통과 항목이 있으면 **해당 Step으로 돌아가서 보완**
- 보완 후 다시 검증

---

## Step 7: ADR 작성

자기 검증을 통과하면 ADR 문서를 자동 생성:
1. `docs/adr/` 디렉토리에 ADR 파일 생성
2. 번호 자동 부여 (기존 ADR 번호 확인)
3. 아래 ADR 템플릿에 따라 작성

---

## Step 8: 확인

1. 설계 요약 제시
2. ADR 문서 경로 제시
3. 자기 검증 결과 제시 (체크리스트 통과 여부)
4. 다음 단계 안내 (`/tdd`로 TDD 구현 시작)
5. **CONFIRM** 대기

## 설계 흐름 원칙

```
컨텍스트 수집 (자동, Silent)
        ↓
현황 공유 → 요구사항 정의
        ↓
용량/규모 추정 (항상 필수)
        ↓
숫자 + 기능 특성 기반으로 논의
        ↓
결정 도출 (선택 근거 명시)
        ↓
접근법 제안 (추론 프레임워크 거친 후)
        ↓
상세 설계
        ↓
자기 검증 (품질 게이트)
        ↓
ADR 작성
```

**핵심**: 규모 추정이 먼저, 그 숫자가 나머지 결정을 이끈다.
요구사항에서 명시적으로 불필요하다고 한 항목은 논의 없이 스킵한다.

## Integration
- `/design` → `/tdd` (설계 후 TDD 구현)
- `/design` → `/scaffold` (새 도메인이면 보일러플레이트 먼저 생성)
- `/design` → `/code-review` (구현 후 리뷰)
- `/orchestrate feature` 내에서 자동 호출 가능
